# Purchase Flow: Cart → Checkout → Payment → Order

This document describes the end-to-end purchase lifecycle implemented in the backend:
- domain responsibilities
- service-level business rules
- allowed state transitions
- HTTP endpoints and response conventions

This project follows strict layered architecture:
**routes → controllers → services** (business rules live in services only).

---

## Response & Error Conventions

### Standard success response
All successful HTTP responses use:

```json
{
  "success": true,
  "message": "Some message",
  "data": {}
}
```

Implemented by `utils/response.success(res, data, message)`.

### Standard error response

All error HTTP responses use:

```json
{
  "success": false,
  "message": "Some error message"
}
```

Errors are thrown from services using `AppError(message, statusCode)` and propagated via `next(err)` to the centralized error middleware.

## Domain Overview

### 1) Cart

**Purpose:** The cart is the mutable workspace where the user builds the purchase:
- add/update/remove items
- maintain a deterministic summary
- store market/currency metadata needed for display and checkout rules

**Key characteristics:**
- Identified by `cartId` generated by backend
- Not shared across users (private cart)
- Contains:
  - `items[]`
  - `summary` (itemsCount, subtotalUSD)
  - `metadata` (market, currencies, exchange rate, tax, customer, status, timestamps)

**Cart metadata (current schema highlights):**
- `market`: "VE"
- `baseCurrency`: "USD"
- `displayCurrency`: "USD" | "VES"
- `exchangeRate`:
  - `provider`: "BCV"
  - `usdToVes`: number | null
  - `asOf`: string | null
- `tax`:
  - `priceIncludesVAT`: true
  - `vatRate`: 0.16
- `customer`: { email, name, phone }
- `status`: "active" | "locked" | "checked_out"
- `createdAt` / `updatedAt`

**Cart status intent:**
- `active`: cart can be modified and used for checkout
- `locked`: cart should not be modified (payment in progress / reserved)
- `checked_out`: terminal (cart completed)

---

### 2) Checkout

**Purpose:** Checkout is a validated snapshot of a cart, responsible for:
- verifying cart can be checked out
- revalidating stock at checkout time
- computing final totals in USD and (optionally) VES using exchange rate
- exposing allowed payment methods

**Checkout does not:**
- charge money
- integrate with payment providers
- create orders

**Checkout service API (implemented):**
- `createCheckout(cartId)`
- `getCheckoutById(checkoutId)`

**Checkout data model:**

```json
{
  "checkoutId": "string",
  "cartId": "string",
  "totals": { "subtotalUSD": 0, "subtotalVES": 0 | null },
  "exchangeRate": { "provider": "BCV", "usdToVes": 0, "asOf": "string" } | null,
  "paymentMethods": { "usd": ["zelle","zinli"], "ves": ["bank_transfer","pago_movil"] }
}
```

---

### 3) Payment

**Purpose:** Payment tracks the chosen payment method and its lifecycle, including customer submission and admin review.

This MVP supports manual payment flows common in Venezuela:

**USD methods:**
- `zelle`
- `zinli`

**VES methods:**
- `pago_movil`
- `bank_transfer`

**Payment service API (implemented):**
- `createPayment(checkoutId, method)`
- `submitPayment(paymentId, proof)`
- `confirmPayment(paymentId, note)` (admin)
- `rejectPayment(paymentId, reason)` (admin)

**Payment statuses (current):**
- `pending` → created, waiting for customer proof
- `submitted` → customer submitted proof
- `confirmed` → admin confirmed payment
- `rejected` → admin rejected payment

**Payment model (current):**

```json
{
  "paymentId": "string",
  "checkoutId": "string",
  "method": "zelle|zinli|pago_movil|bank_transfer",
  "currency": "USD|VES",
  "amount": 0,
  "status": "pending|submitted|confirmed|rejected",
  "proof": { "reference": "string" } | null,
  "review": { "note": "string|null" } | { "reason": "string" },
  "createdAt": "ISO string",
  "updatedAt": "ISO string"
}
```

---

### 4) Order (next domain)

**Purpose:** An Order is the immutable record of a completed purchase.

**Core rule (to implement next):**
> Create an Order only after a Payment is confirmed.

Orders will later drive:
- fulfillment
- inventory adjustments
- customer notifications
- refunds

## Flow Summary

### Happy path
1. Create Cart
2. Add items / update metadata
3. Create Checkout from cart
4. Create Payment from checkout (pending)
5. Customer submits proof (submitted)
6. Admin confirms payment (confirmed)
7. **Create Order** (created)

### Rejected payment path
- Payment submitted → admin rejects → payment becomes `rejected`
- User may retry payment (policy to define in Orders step / Payment retry rules)

---

## Allowed State Transitions

### Cart (intent)
- `active` → may later become `locked` (during payment flow)
- `locked` → may later become `checked_out` (after order completion)

**Note:** Cart state transitions may be enforced more strictly as Orders are introduced.

### Payment (enforced)
- `pending` → `submitted` (customer submits proof)
- `submitted` → `confirmed` (admin confirms)
- `submitted` → `rejected` (admin rejects)

**Disallowed:**
- `pending` → `confirmed`/`rejected`
- `confirmed`/`rejected` → any other state (terminal in MVP)

## Service-Level Business Rules (Source of Truth)

### Checkout: `createCheckout(cartId)`

- **404** if cart does not exist (propagated from cart service)
- **400** "Cart is empty" if cart has no items
- **409** "Cart is not active" if `cart.metadata.status !== "active"`
- Revalidate stock for each cart item:
  - if `!product.inStock || product.stock <= 0 || item.quantity > product.stock`
  - throw **409** "Insufficient stock"
- Totals:
  - `subtotalUSD` = cart.summary.subtotalUSD
  - `subtotalVES` computed only if:
    - `usdToVes` is a number > 0
    - `asOf` is a non-empty string
  - otherwise `subtotalVES = null` and `exchangeRate = null`
- Payment methods are always returned:
  - `usd`: ["zelle","zinli"]
  - `ves`: ["bank_transfer","pago_movil"]

---

### Payments: `createPayment(checkoutId, method)`

- `checkout` is retrieved via `checkoutService.getCheckoutById(checkoutId)`
- **404** "Checkout not found" if missing
- **400** "Invalid payment method" if method not in:
  - `zelle`, `zinli`, `pago_movil`, `bank_transfer`
- If method is USD (`zelle`|`zinli`):
  - `currency = "USD"`
  - `amount = checkout.totals.subtotalUSD`
- If method is VES (`pago_movil`|`bank_transfer`):
  - requires `checkout.totals.subtotalVES !== null`
  - otherwise **400** "Exchange rate required"
  - `currency = "VES"`
  - `amount = checkout.totals.subtotalVES`
- Creates payment in `pending` with `proof = null`

---

### Payments: `submitPayment(paymentId, proof)`

- **404** "Payment not found" if missing
- **409** "Payment is not pending" if `status !== "pending"`
- **400** "Invalid payment proof" if:
  - `proof` is missing OR
  - `proof.reference` is not a non-empty string
- On success:
  - `status = "submitted"`
  - `proof` stored
  - `updatedAt` changes (monotonic update strategy used to avoid flakiness)

---

### Payments Admin: `confirmPayment(paymentId, note)`

- **404** "Payment not found" if missing
- **409** "Payment is not submitted" if `status !== "submitted"`
- `note`:
  - optional
  - if provided must be non-empty string after trim
  - else **400** "Invalid payment review"
- On success:
  - `status = "confirmed"`
  - `review = { note: string|null }`
  - `updatedAt` changes

---

### Payments Admin: `rejectPayment(paymentId, reason)`

- **404** "Payment not found" if missing
- **409** "Payment is not submitted" if `status !== "submitted"`
- `reason`:
  - required non-empty string after trim
  - else **400** "Invalid payment review"
- On success:
  - `status = "rejected"`
  - `review = { reason: string }`
  - `updatedAt` changes

---

## HTTP Endpoints (Current)

### Cart
- `POST /api/cart`
- `GET /api/cart/:cartId`
- `POST /api/cart/:cartId/items`
- `PATCH /api/cart/:cartId/items/:productId`
- `DELETE /api/cart/:cartId/items/:productId`
- `PATCH /api/cart/:cartId/metadata`

### Checkout
- `POST /api/checkout`
- (internal access supported) `checkoutService.getCheckoutById(checkoutId)` is available at service level

### Payments
- `POST /api/payments`
- `PATCH /api/payments/:paymentId/submit`
- `PATCH /api/payments/:paymentId/confirm`
- `PATCH /api/payments/:paymentId/reject`

## Input Validation Summary

| Endpoint | Field | Validation | Error |
|----------|-------|------------|-------|
| `POST /api/payments` | `checkoutId` | Required | 404 if checkout not found |
| `POST /api/payments` | `method` | Must be: `zelle`, `zinli`, `pago_movil`, `bank_transfer` | 400 "Invalid payment method" |
| `POST /api/payments` | `method` (VES) | Requires `checkout.totals.subtotalVES !== null` | 400 "Exchange rate required" |
| `PATCH /:id/submit` | `proof` | Required object | 400 "Invalid payment proof" |
| `PATCH /:id/submit` | `proof.reference` | Required, non-empty string | 400 "Invalid payment proof" |
| `PATCH /:id/submit` | status | Must be `pending` | 409 "Payment is not pending" |
| `PATCH /:id/confirm` | `note` | Optional, if present must be non-empty after trim | 400 "Invalid payment review" |
| `PATCH /:id/confirm` | status | Must be `submitted` | 409 "Payment is not submitted" |
| `PATCH /:id/reject` | `reason` | Required, non-empty string after trim | 400 "Invalid payment review" |
| `PATCH /:id/reject` | status | Must be `submitted` | 409 "Payment is not submitted" |
| `POST /api/checkout` | cart | Must exist | 404 if cart not found |
| `POST /api/checkout` | cart.items | Must have at least 1 item | 400 "Cart is empty" |
| `POST /api/checkout` | cart.metadata.status | Must be `"active"` | 409 "Cart is not active" |
| `POST /api/checkout` | cart.items[].quantity | Must not exceed product.stock | 409 "Insufficient stock" |

## Why This Documentation Matters

This documentation is the "domain contract" that prevents accidental coupling:
- **Cart** is mutable and user-driven
- **Checkout** is a validated snapshot for payment intent
- **Payment** is the auditable lifecycle for manual/online payment steps
- **Order** will be the immutable sales record

This makes future work safer:
- moving to DB (swap Maps for repos via DI)
- integrating real payment providers
- adding authentication/admin roles
- building a frontend that matches backend truth
